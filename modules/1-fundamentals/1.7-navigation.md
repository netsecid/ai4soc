## The ai4soc Command Reference

All custom ai4soc commands are prefixed with `/course-` to avoid conflicts
with Claude Code's built-in commands. SOC workflow triggers use natural
language prefixes.

### Course Navigation Commands

| Command | What It Does |
|---------|-------------|
| `/course-start` | Begin the course from Module 0.0 |
| `/course-check` | Show your current progress through the modules |
| `/course-restart` | Reset progress and begin from the beginning |

### SOC Workflow Triggers

These are not strict commands — they are natural language triggers that
activate the relevant workflow and role context. Always follow them with
what you are working on.

| Trigger | Activates | Example |
|---------|-----------|---------|
| `/triage [file or description]` | SOC Analyst mode + triage workflow | `/triage lab/alerts/scenario-01-alert-bruteforce.json` |
| `/ir [incident description]` | Incident Responder mode | `/ir Active RDP session from Tor IP on WKSTN-FIN-042` |
| `/cti [report, IOC, or actor]` | CTI Analyst mode | `/cti 185.220.101.47 — what do we know about this actor` |
| `/detect [behavior]` | Detection Engineer mode | `/detect PowerShell spawned from Office process` |
| `/report [period or incident]` | SOC Reporter mode | `/report scenario-01 executive summary` |
| `/enrich [indicator]` | IOC enrichment workflow | `/enrich 185.220.101.47` |
| `/sigma [behavior]` | Sigma rule generation | `/sigma RDP brute force followed by successful login` |
| `/attack [technique]` | MITRE ATT&CK lookup | `/attack T1110 — brute force` |

### Claude Code Built-in Commands (Reserved)

These belong to Claude Code itself — do not try to use them for ai4soc workflows:

| Command | What It Does |
|---------|-------------|
| `/status` | Show session info, model, MCP connections |
| `/help` | Show Claude Code help |
| `/clear` | Clear the conversation |
| `/exit` | Exit Claude Code |

## Navigation Patterns

### Starting a New Session

Always start from inside the `ai4soc/` directory:

```bash
cd C:\Users\iyenk\VSCode\ai4soc    # Windows
cd ~/ai4soc                          # macOS / Linux

claude
```

First message of every session — confirm context loaded:

```
What SOC environment am I working in?
```

If the response describes PT Nusantara Digital and the lab stack, you are
ready. If it gives a generic answer, `CLAUDE.md` did not load — check your
working directory.

### Resuming After a Break

Context does not persist between Claude Code sessions. When you return after
closing and reopening:

1. Start Claude Code from inside `ai4soc/`
2. Confirm context loaded
3. Orient the agent to where you left off:

```
I was working on the scenario-01 incident investigation.
Read the following files to catch up on what we found:
- outputs/[date]/triage-scenario-01.md
- outputs/[date]/ir-response-scenario-01.md

Summarize where we left off and what still needs to be done.
```

The outputs folder is your session memory between conversations.

### Switching Between Tasks

When moving from one task to another within a session, be explicit:

```
We are done with the triage on scenario-01.
Switching to Detection Engineer mode.
Now let's focus on writing a Sigma rule for the RDP brute force pattern.
```

This prevents the agent from mixing context between different tasks.

Read all files in outputs/[today's date]/ and give me a summary of

## Exercise 2 — @-Mentions for Context

Claude Code supports `@` references to pull specific file content into context
explicitly. This is useful when you want the agent to focus on a specific file
without reading everything:

```
@lab/alerts/scenario-02-alert-phishing.json

Focus only on the BEC email alert (the third alert in the chain).
What is the exact amount being requested and what bank account details
are provided? Does anything about the request look suspicious beyond
the source context?
```

Use `@` mentions when:
- You want the agent focused on one specific file
- You are referencing a file you have not discussed yet in this session
- You want to contrast two specific files

## Building Good Agent Habits

After one full module of practice, here are the habits that consistently
produce better outputs:

**Be specific about what you want.** Vague prompts produce vague outputs.
"Analyze this alert" is less useful than "Read this alert, identify the MITRE
techniques, enrich the source IP, and give me a triage verdict with confidence
level and justification."

**Always specify output location.** "Save this" without a path leaves the
agent guessing. "Save to outputs/2025-01-14/triage-scenario-01.md" is unambiguous.

**Correct immediately and precisely.** When the agent makes a mistake, correct
it in the same message: "That is wrong — the successful login (Event ID 4624)
happened at 02:11, not 02:03. Please revise the timeline."

**State role transitions explicitly.** "Switching to Detection Engineer mode now"
produces a cleaner context shift than letting the agent guess.

**Review every output before filing.** The agent produces fast, consistent output.
Consistent is not the same as correct. Always read it before treating it as final.

**Use the outputs folder as your memory.** Outputs persist across sessions.
Reference them at the start of each new session to restore context.
