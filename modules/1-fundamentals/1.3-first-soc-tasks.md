# 1.3 First SOC Tasks — Triage, Enrich, Document

This lesson puts everything together into the three core tasks that make up the
majority of a SOC analyst's day: triaging an alert, enriching the indicators,
and documenting the findings. You have touched these individually — now you run
the full workflow end to end.

---

## The Full Triage Workflow

A complete alert triage has five steps. Every exercise in this lesson follows
this sequence:

```
1. INTAKE       Read the alert, understand what fired and why
2. ENRICH       Look up IOCs, asset context, user history
3. CORRELATE    Check for related alerts, historical patterns
4. DECIDE       True Positive / False Positive / Needs Investigation
5. DOCUMENT     Write it up, save it, file it
```

By the end of this lesson you will have run this workflow twice — once for
each scenario in the lab — and produced two complete triage reports.

## Workflow 1 — Scenario 01: RDP Brute Force

### Step 1 — Intake

```
Read lab/alerts/scenario-01-alert-bruteforce.json

Give me an intake summary:
- What rule fired and at what severity?
- What asset and user are involved?
- What is the source of the activity?
- How many alerts are in this chain and what is the sequence?
```

Read the agent's response. Confirm it has correctly identified the three-alert
chain: brute force → successful login → directory enumeration.

### Step 2 — Enrich

```
Now enrich the key indicators from this alert:

1. Source IP 185.220.101.47 — what do we know about this?
2. Destination asset WKSTN-FIN-042 — what department owns it and 
   what is its risk profile given the organizational context in CLAUDE.md?
3. User budi.santoso — what is their role and what data would they 
   legitimately have access to?
4. The RDP service being targeted — is RDP exposure expected for 
   this type of asset?
```

### Step 3 — Correlate

```
Based on the alert data and the IOC list at lab/ioc/scenario-01-02-ioc-list.csv:

1. Does the source IP appear in any other alerts or IOC data?
2. What does the post-login behavior tell us about the attacker's intent?
3. What should we check next to understand the full scope?
```

The agent should identify that `185.220.101.47` appears in both scenarios —
this is a key finding. If it misses this, prompt it:

```
Check the IOC list for whether 185.220.101.47 appears in multiple scenarios.
What does that tell us?
```

### Step 4 — Decide

```
Based on the intake, enrichment, and correlation:

What is your verdict on this alert chain?
- True Positive, False Positive, or Needs Investigation?
- Confidence level: High, Medium, or Low?
- One sentence justification.

What is the urgency? Is this still active or is it resolved?
```

### Step 5 — Document

```
Using the Alert Triage Summary template from CLAUDE.md, produce a complete
triage report for scenario-01.

Include all findings from our intake, enrichment, and correlation steps.
Save it to outputs/[today's date]/triage-scenario-01.md
```

Open the file. Read every section. This is your deliverable — treat it as
something you would actually submit.

## Review Your Outputs

At this point you should have the following files in your outputs folder:

```
outputs/[today's date]/
├── triage-scenario-01.md
├── triage-scenario-02.md
├── campaign-assessment-jan2025.md
└── notification-scenario-02-bec.md
```

Open each one and review it as if you were a senior analyst reviewing a
junior's work. Ask yourself:

- Is the verdict clearly justified by the evidence?
- Are there any claims that are not backed by data from the alert or logs?
- Would someone reading this without context understand what happened?
- Are the recommended actions specific enough to act on?

If anything is wrong or weak, tell the agent to fix it:

```
In triage-scenario-01.md, the "Actions Taken" section is too vague.
Make each action item specific — what exactly should be done, by whom, 
and in what order?
```

## What Good Triage Output Looks Like

A triage report is useful when:

**It stands alone.** Someone who was not in the analysis session can read it
and understand what happened, what the verdict is, and what was done.

**Evidence backs every claim.** Every assertion — "this IP is malicious",
"the login is anomalous", "this is a True Positive" — should point to specific
evidence from the alert or log data.

**Uncertainty is named.** If something is unclear or requires further investigation,
the report says so explicitly. A good analyst writes "could not confirm X —
recommend checking Y" rather than guessing.

**Actions are specific.** Not "investigate further" — but "check whether session
0x3e7f2a is still active on WKSTN-FIN-042 and isolate the machine if confirmed."
