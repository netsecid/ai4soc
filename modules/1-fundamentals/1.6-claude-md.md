# 1.6 CLAUDE.md — Giving Your Agent Persistent SOC Context

`CLAUDE.md` is the most important file in the entire ai4soc project. Every
other capability you have learned — file operations, parallel agents, role
switching — depends on the agent having the right context to work with.

This lesson teaches you how `CLAUDE.md` works, why it matters, and how to
customize it for your own environment when you move beyond the lab.

---

## What CLAUDE.md Actually Does

Every time you start Claude Code inside the `ai4soc/` directory, it reads
`CLAUDE.md` before your first message. This means the agent begins every
session already knowing:

- What organization it is protecting
- What security tools are in use
- What the threat profile looks like
- What output templates to use
- What roles are available and how to switch between them
- What the lab file structure looks like and where to write outputs

Without `CLAUDE.md`, the agent is a blank slate. With it, the agent is a
colleague who has already been briefed.

This is what distinguishes a well-configured AI agent from a generic chatbot.
The context is the configuration.

---

## Anatomy of CLAUDE.md

Open `CLAUDE.md` in VS Code alongside this lesson. Walk through each section:

### Disclaimers
Two blocks at the top — education purpose and data sanitization. These exist
so that anyone who picks up this file and starts working with it immediately
understands the safety guardrails. Do not remove these when customizing for
your own environment.

### Who You Are
Defines the agent's behavioral identity — a SOC analyst assistant, not a
generic AI. Sets ground rules: support the analyst's judgment, never fabricate
IOCs, flag real data. This section shapes every response the agent produces.

### Organization Context
The fictional org details for the lab. When you move to a real environment,
this is where you describe your actual organization — industry, region,
compliance requirements, threat profile, and crown jewels.

### Security Stack
Lists the tools in the environment. The agent uses this to give tool-specific
advice. If you tell it you use Datadog SIEM instead of Wazuh, its detection
rule advice will shift accordingly.

### Lab Environment Structure
Maps the folder structure so the agent knows where to read from and write to.
This section is especially important for file operations — the agent references
it when deciding where to save outputs.

### Role Definitions and Flow
The five roles and the flow diagram showing when to switch. This is the
analytical framework the agent uses when you activate a role explicitly.

### Slash Commands
The command reference the agent uses to understand workflow triggers.
Note the `[CUSTOMIZE]` markers — these are the sections you modify when
adapting for a real environment.

### Output Templates
Standardized formats for every artifact type. These ensure consistency
across all outputs regardless of which analyst or session produced them.

### Enrichment Order
The sequence of sources to check for each IOC type. The agent follows
this order when you ask for IOC enrichment.

### OpSec Rules
Behavioral guardrails that apply to every session. These are non-negotiable
— they exist to protect you and your organization.

## Exercise 1 — Understand What the Agent Knows

Before you can customize `CLAUDE.md`, you need to understand exactly what
the agent reads from it. Ask it:

```
Without referencing any alert or log files, tell me everything you know 
about the organization you are protecting in this lab environment.

Include: org name, industry, region, compliance requirements, threat profile,
security stack, and crown jewels.
```

The agent should recite the PT Nusantara Digital profile accurately. If it
gets anything wrong or makes something up, that is a sign of a gap in
`CLAUDE.md` that needs to be filled.

---

## Exercise 2 — Test the Limits

Find where the agent's context ends:

```
I need to triage an alert from our Palo Alto firewall. What fields 
should I look for and how do they map to our threat profile?
```

The agent should acknowledge that Palo Alto firewall is not in the lab
security stack defined in `CLAUDE.md`. A well-configured agent handles
this gracefully — it can still help, but it flags that this tool is not
in its known context and its advice is therefore more generic.

This shows you what happens when the context and the real environment diverge —
and why keeping `CLAUDE.md` accurate matters.

## Exercise 3 — Modify CLAUDE.md for a Custom Stack

Practice adding a tool to the security stack. Open `CLAUDE.md` in VS Code
and find the Security Stack section:

```
SIEM:         Wazuh 4.x              ← Primary lab SIEM
EDR:          CrowdStrike Falcon      ← [CUSTOMIZE]
NETWORK:      Zeek + Suricata
CLOUD:        AWS (primary), GCP (secondary)
TICKETING:    Jira (simulated)
COMMS:        Slack (simulated)
THREAT_INTEL: VirusTotal, AbuseIPDB, Shodan, MISP (free tiers)
```

Add a line for a SOAR platform:

```
SOAR:         Shuffle (open source)   ← [CUSTOMIZE]
```

Save the file. Then restart Claude Code and ask:

```
What SOAR platform is in our security stack and how does it integrate 
with our SIEM for automated response?
```

The agent should now reference Shuffle. This demonstrates the live link between
`CLAUDE.md` content and agent behavior — change the file, change the agent.

## Exercise 4 — Add a Custom Output Template

You will often need output formats that are specific to your organization —
a custom ticket format, a specific report structure, a partner notification
template. You add these directly to `CLAUDE.md`.

Add a custom template for a threat hunting hypothesis:

```markdown
### 5. Threat Hunting Hypothesis
\```
HUNTING HYPOTHESIS
==================
Hypothesis ID   : [HYP-YYYY-NNN]
Created         : [YYYY-MM-DD]
Analyst         : [name]
Status          : [Active / Validated / Closed]

HYPOTHESIS STATEMENT
--------------------
[One sentence: "I believe [threat actor/behavior] is occurring because [signal/indicator]"]

BASED ON
--------
[What triggered this hypothesis — alert, CTI report, anomaly, intuition]

MITRE ATT&CK
------------
Tactic     : [tactic]
Technique  : [TXXXX — name]

HUNT QUERIES
------------
[The actual queries you will run — KQL, SPL, DSL, or hunting steps]

SUCCESS CRITERIA
----------------
[What would confirm the hypothesis — what evidence would you expect to find]

RESULTS
-------
[Fill in after hunting — what did you find]

VERDICT
-------
[Confirmed / Refuted / Inconclusive]
\```
```

Save `CLAUDE.md`. Then test it:

```
Create a threat hunting hypothesis for potential lateral movement from 
WKSTN-FIN-042 following the scenario-01 incident.
Use the Threat Hunting Hypothesis template from CLAUDE.md.
Save to outputs/[today's date]/hunt-hypothesis-lateral-movement.md
```

The agent should now use your custom template.

## Adapting CLAUDE.md for a Real Environment

When you are ready to use ai4soc workflows in your actual SOC, follow this
checklist to customize `CLAUDE.md`:

**Organization context:**
- [ ] Replace "PT Nusantara Digital" with your org name (or keep it generic if confidential)
- [ ] Update industry, region, and compliance requirements
- [ ] Update the threat profile to match your actual adversary landscape
- [ ] Update crown jewels to reflect your real high-value assets

**Security stack:**
- [ ] Update SIEM to your actual platform (Datadog, Splunk, Elastic, QRadar, etc.)
- [ ] Update EDR to your actual platform
- [ ] Add any tools specific to your environment
- [ ] Remove tools you do not use

**Output templates:**
- [ ] Adapt the triage template to match your ticketing system fields
- [ ] Update the SIR template to match your incident reporting format
- [ ] Add any org-specific templates (compliance reports, partner notifications, etc.)

**OpSec rules:**
- [ ] Add any org-specific data handling requirements
- [ ] Add any compliance-specific restrictions
- [ ] Update enrichment sources to include your licensed threat intel feeds

**File structure:**
- [ ] Update paths if your folder structure differs from the lab

> **Important:** When adapting for a real environment, be careful about what
> you include in `CLAUDE.md`. Do not put real credentials, real system
> hostnames, real employee names, or other sensitive identifiers in this file —
> especially if you use cloud AI tiers where the file content is sent to an API.
> Keep `CLAUDE.md` at the level of role and tool descriptions, not operational details.

---

## CLAUDE.md Is a Living Document

Your `CLAUDE.md` will evolve. As you learn more about what helps the agent
produce better outputs, you will refine it. As your environment changes, you
will update it. As you add new templates or workflows, you will extend it.

Treat it like a runbook — review it periodically, keep it accurate, and
commit changes to version control so you have a history of how it evolved.

---

## Key Takeaways

`CLAUDE.md` is your SOC configuration file for the AI agent. Everything the
agent knows about your environment, your tools, your workflows, and your
expected outputs comes from this file.

Understanding how to read, extend, and adapt it is one of the most transferable
skills in this course — because the same principle applies to any agentic AI
system you will use in a real SOC environment.
