# 1.4 Parallel Agents — Triage 10 Alerts Simultaneously

A SOC analyst on a busy shift does not get alerts one at a time. They land in
batches. During a peak period or an active incident, the queue fills faster than
a single analyst can triage sequentially.

Parallel agents change this equation fundamentally.

---

## What Parallel Agents Are

When you ask Claude Code to process multiple things simultaneously, it spawns
independent subagent instances — each one working on a separate task at the
same time. These subagents share no conversation history with each other, but
they all have access to the same files and the same `CLAUDE.md` context.

The result: work that would take 50 minutes sequentially takes 6–8 minutes
in parallel.

```
Sequential (no AI):
Alert 1 → 10 min → Alert 2 → 10 min → Alert 3 → 10 min → Total: 30 min

Sequential (with AI, one at a time):
Alert 1 → 5 min → Alert 2 → 5 min → Alert 3 → 5 min → Total: 15 min

Parallel (with AI agents):
Alert 1 ─┐
Alert 2 ──┼─ all running simultaneously ─→ Total: ~6 min
Alert 3 ─┘
```

---

## When to Use Parallel Agents

Parallel agents are most valuable when:

- You have multiple independent alerts to triage (no dependency between them)
- You need to process a batch of IOCs for enrichment simultaneously
- You want to analyze the same scenario from multiple role perspectives at once
- You are generating multiple detection rules for a related set of behaviors

Parallel agents are less useful when:

- Each task depends on the output of the previous one (use sequential instead)
- You are doing a single deep investigation that requires building context
  across steps

## Exercise 1 — Parallel Triage of Both Scenarios

Instead of triaging scenario-01 and scenario-02 one after the other, ask the
agent to do both simultaneously:

```
I need you to triage two alerts in parallel simultaneously.

For EACH alert, run the full triage workflow independently:
- Read the alert file
- Analyze the attack chain
- Enrich the key IOCs
- Make a verdict with confidence level and justification
- Save a complete triage report using the Alert Triage Summary template from CLAUDE.md

Alert 1: lab/alerts/scenario-01-alert-bruteforce.json
Save report to: outputs/[today's date]/parallel-triage-scenario-01.md

Alert 2: lab/alerts/scenario-02-alert-phishing.json  
Save report to: outputs/[today's date]/parallel-triage-scenario-02.md

Run both tasks at the same time and tell me when both are complete.
```

Watch what happens in the terminal. You will see Claude Code spawn two
subagents working simultaneously. Both reports appear in your `outputs/`
folder when complete.

## Exercise 3 — Multi-Perspective Analysis

This is one of the most powerful applications of parallel agents for SOC work:
analyzing the same incident from multiple role perspectives at the same time.

```
Analyze scenario-01 from three different perspectives simultaneously.
Each subagent should read lab/alerts/scenario-01-alert-bruteforce.json
and lab/logs/windows-event-scenario-01.json, then write their analysis
from their specific role perspective:

Subagent 1 — SOC Analyst perspective:
Focus on triage verdict and immediate actions needed.
Save to: outputs/[today's date]/perspective-analyst-scenario-01.md

Subagent 2 — Detection Engineer perspective:
Focus on what detection gaps this incident reveals and what rules
should be written to catch this behavior in the future.
Save to: outputs/[today's date]/perspective-detection-scenario-01.md

Subagent 3 — CTI Analyst perspective:
Focus on threat actor profiling, TTP mapping to MITRE ATT&CK,
and what this activity tells us about adversary intent.
Save to: outputs/[today's date]/perspective-cti-scenario-01.md

Run all three simultaneously.
```

When complete, compare the three outputs. Each subagent approached the same
data with a different lens. Together they give you a more complete picture
than any single analysis would.

This is a real-world capability: in a large SOC, you might have an analyst,
a detection engineer, and a CTI analyst all working the same incident from
different angles. With parallel agents, one person can produce all three
perspectives simultaneously.

## Understanding the Limits of Parallelism

**Subagents do not share context with each other.** Each parallel subagent
starts fresh with only `CLAUDE.md` and the files you point it at. If one
subagent discovers something critical, the others will not know about it
unless you explicitly feed that finding back in.

This is why the synthesis step matters. The parallel subagents produce raw
material — your job is to review and combine their outputs into a coherent
whole.

**Too many parallel tasks degrades quality.** Running 20 parallel agents
at once is technically possible but each one gets less careful. For complex
analysis tasks, 3–5 parallel agents is a practical ceiling. For simpler
tasks like IOC enrichment, you can go higher.

**Parallel outputs need individual review.** Do not assume that because
four agents ran simultaneously, their outputs are all correct. Each one
needs the same scrutiny you would apply to a sequential output.

## Practical SOC Application

The real-world pattern for using parallel agents in a busy SOC shift:

```
Morning shift starts. 23 alerts in the queue.

Step 1: Quick sort
Ask the agent to read all 23 alerts (batch summary, not deep analysis)
and categorize by type and severity. Takes 3 minutes.

Step 2: Parallel deep triage
Take the top 5 High/Critical alerts. Run them in parallel.
Takes 8 minutes instead of 25.

Step 3: Sequential handling for medium alerts
The remaining 18 Medium/Low alerts — run 5 at a time in parallel
batches. Covers all 18 in ~20 minutes instead of 90.

Step 4: Synthesis and escalation
Review parallel outputs, escalate confirmed TPs, close confirmed FPs,
document everything.

Total shift time saved: 60-70 minutes on alert triage alone.
```
