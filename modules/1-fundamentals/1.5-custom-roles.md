# 1.5 Custom Roles — Analyst, Responder, Hunter, Engineer

You have seen the role flow defined in `CLAUDE.md`. This lesson turns those
role definitions into practice. You will learn how to activate each role
explicitly, understand when to switch between them, and use them together
in a realistic incident scenario.

---

## Why Roles Matter

Every role in `CLAUDE.md` represents a different way of looking at the same
data. The same alert that a SOC Analyst reads as a triage question, a Detection
Engineer reads as a coverage gap, and a CTI Analyst reads as an adversary
behavior signal.

Training yourself to switch between these lenses — and to know when to switch —
is one of the most valuable skills in security operations. The AI roles are
a scaffold for building that habit.

---

## How to Activate a Role

You activate a role by telling the agent explicitly which perspective to take.
The more specific you are, the more useful the output:

**Implicit (works, but vague):**
```
Help me investigate this alert.
```

**Explicit role activation (better):**
```
Shift into Detection Engineer mode.
Read lab/alerts/scenario-01-alert-bruteforce.json and tell me what 
detection gaps this incident reveals. What rules should exist but don't?
```

**Full role context (best):**
```
You are now in Detection Engineer mode as defined in CLAUDE.md.
Your focus is on detection coverage gaps and rule quality.

Read lab/alerts/scenario-01-alert-bruteforce.json.

Tell me:
1. What MITRE ATT&CK techniques does this attack use?
2. Which techniques have detection coverage in Wazuh already?
3. Which techniques have no current detection?
4. What is the highest priority gap to close first?
```

The more clearly you specify the role's focus, the sharper the output.

Work through each role using scenario-01 as the base incident. By the end

### 🔵 SOC Analyst — Triage and Scope

This is your default starting point. You have already practiced this in 1.3.
Use it here to set the baseline for the other roles.

```
SOC Analyst mode.

Read lab/alerts/scenario-01-alert-bruteforce.json.

What is the current scope — how many systems, users, and data types
are confirmed affected at this point in the investigation?

Save a scope summary to:
outputs/[today's date]/roles-01-analyst-scope.md
```

### 🔴 Incident Responder — Containment and Timeline

Switch into IR mode after the analyst confirms a True Positive.

```
Incident Responder mode.

The scenario-01 alert has been confirmed as a True Positive.
A threat actor successfully authenticated via RDP to WKSTN-FIN-042
and enumerated Finance folders.

Read:
- lab/alerts/scenario-01-alert-bruteforce.json
- lab/logs/windows-event-scenario-01.json

Your focus: what happens in the next 30 minutes.

Produce:
1. A prioritized containment action list — what must happen first, second, third
2. A reconstruction of the attack timeline in chronological order
3. A list of open questions that need answers before you can confirm eradication
4. A draft stakeholder notification using the Alert Notification template from CLAUDE.md

Save to: outputs/[today's date]/roles-02-ir-response.md
```

**Key things to look for in the output:**
- Does it prioritize isolating WKSTN-FIN-042 before other actions?
- Does it flag checking whether the session is still active?
- Does it flag that Finance data may have been accessed/exfiltrated?
- Does it recommend resetting budi.santoso's credentials?

### 🟡 CTI Analyst — Adversary Profiling

```
CTI Analyst mode.

Read:
- lab/alerts/scenario-01-alert-bruteforce.json
- lab/alerts/scenario-02-alert-phishing.json
- lab/ioc/scenario-01-02-ioc-list.csv

Both scenarios share the same C2 infrastructure (185.220.101.47).
Your job is to profile the threat actor behind this activity.

Produce:
1. MITRE ATT&CK technique mapping for both scenarios combined
2. An adversary profile — what do their TTPs tell us about their capability,
   intent, and sophistication level?
3. What is their likely next move if they still have access?
4. What additional IOCs should we hunt for based on this profile?
5. What threat actor groups or campaigns does this pattern resemble?

Save to: outputs/[today's date]/roles-03-cti-profile.md
```

**Note on attribution:** The agent should be careful here. It should present
patterns and similarities — not claim definitive attribution without evidence.
Good CTI analysis is probabilistic. If the agent states attribution as fact,
correct it:

```
Do not state attribution as confirmed. Rephrase as "consistent with" or 
"similar TTPs to" — attribution requires additional corroborating evidence.
```

### 🟢 Detection Engineer — Coverage and Rule Writing

```
Detection Engineer mode.

Read lab/alerts/scenario-01-alert-bruteforce.json and 
lab/logs/windows-event-scenario-01.json.

Map the attack to MITRE ATT&CK techniques and assess detection coverage:

For each technique identified:
1. Is there currently a Wazuh rule that would catch this? (reference the rule IDs in the alert)
2. If yes — is the rule well-tuned or prone to false positives?
3. If no — what would a detection rule look like?

Then write a Sigma rule for the most critical detection gap you find.
Use the Sigma Rule template from CLAUDE.md.

Save the full coverage assessment and Sigma rule to:
outputs/[today's date]/roles-04-detection-coverage.md
```

### 📊 SOC Reporter — Executive Communication

```
SOC Reporter mode.

Read:
- outputs/[today's date]/roles-01-analyst-scope.md
- outputs/[today's date]/roles-02-ir-response.md
- outputs/[today's date]/roles-03-cti-profile.md
- outputs/[today's date]/roles-04-detection-coverage.md

You now have the full picture of this incident from all role perspectives.

Write an executive summary for the CISO and Finance Director of PT Nusantara Digital.

Requirements:
- Maximum one page
- No technical jargon
- Explain what happened in plain language
- State the business impact clearly
- State what was done to respond
- State what is being done to prevent recurrence

```

## Switching Roles Mid-Investigation

In real incidents, you do not follow a clean linear role sequence. You will
often need to switch roles mid-investigation as new information surfaces.

Practice this:

```
I am in the middle of triage on scenario-02 (SOC Analyst mode) and I 
just realized the BEC email is asking for a wire transfer of IDR 487.5M.

Switch to Incident Responder mode immediately.

What are the next three actions I need to take in the next 10 minutes
to prevent financial loss? Be specific and time-ordered.
```

The agent should immediately shift its framing to containment and urgency,
not investigation and analysis. The response should feel different from an
analyst output — faster, more action-oriented, focused on stopping the bleeding.

## Key Takeaways

The role system in ai4soc is not just labels — each role produces a different
type of output, uses different analytical frameworks, and answers different
questions.

Knowing which role to be in at each stage of an incident — and being able to
switch fluidly — is what distinguishes a mature SOC analyst from someone who
just triages tickets.
