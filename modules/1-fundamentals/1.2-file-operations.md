# 1.2 File Operations — Read Logs, Write Reports, Edit Rules

One of the most practical superpowers of an AI coding agent is direct file access.
No copy-paste. No manual formatting. The agent reads your files as they are and
writes its output where you tell it.

This lesson builds that muscle. By the end, you will be comfortable pointing the
agent at any file in the lab and directing where its output goes.

---

## How File Operations Work in Claude Code

Claude Code operates in your current working directory — the folder you started it
from. Since you always start it from inside `ai4soc/`, it has access to everything
in that folder and all subfolders.

You reference files by their relative path:

```
lab/alerts/scenario-01-alert-bruteforce.json
lab/logs/windows-event-scenario-01.json
lab/ioc/scenario-01-02-ioc-list.csv
```

The agent reads these directly. It does not need you to paste the contents.

For output, always direct it to the `outputs/` folder with a dated subfolder:

```
outputs/2025-01-14/triage-scenario-01.md
outputs/2025-01-14/sigma-rule-rdp-bruteforce.yml
outputs/2025-01-14/ioc-enrichment-report.md
```

## Exercise 1 — Read a Log File

Ask the agent to read and summarize the Windows Event log from scenario 01:

```
Read lab/logs/windows-event-scenario-01.json and give me a summary of 
what happened. List the Event IDs present, the timeline of events, and 
identify any anomalies in the sequence.
```

**What to expect:** The agent reads all events in the file and produces a
structured summary. It should identify:
- Event ID 4625 failures (47 total)
- Event ID 4624 successful logon
- Event IDs 4688 and 4663 for process and file access
- The anomaly: successful login from same IP as the failures

**What to check:** Did it get the event count right? Did it correctly identify
the source IP and note the gap between last failure and successful login?

---

## Exercise 2 — Read Multiple Files Together

The agent can read several files in one request and correlate them:

```
Read both lab/alerts/scenario-01-alert-bruteforce.json and 
lab/logs/windows-event-scenario-01.json together.

Tell me what the alert data says versus what the raw log data says.
Are there any discrepancies? Does the raw log add any detail that the 
alert summary missed?
```

**What to expect:** The agent compares the two data sources. It should note that
the raw logs show individual event IDs and timestamps that the alert aggregates
into a count, and that the log includes process execution events (4688) that
give additional post-login context beyond what the alert surface.

This exercise teaches a real analyst skill: cross-referencing alert data with
raw log evidence.

## Exercise 3 — Write a File

Ask the agent to save its analysis:

```
Based on your analysis of the scenario-01 logs and alerts, write a 
concise investigation notes file. Save it to:
outputs/[today's date]/investigation-notes-scenario-01.md

Use plain markdown. Include: timeline, key findings, open questions.
```

After it runs, verify the file was created:

```bash
# In a separate terminal tab (not inside Claude Code)
ls outputs/
```

Open the file in VS Code. Review it. Is it something you would actually
find useful to file with an incident ticket?

## Exercise 4 — Read an IOC List and Enrich It

```
Read lab/ioc/scenario-01-02-ioc-list.csv

For each IOC in the file:
1. Confirm the type (IP, domain, hash, URL, email)
2. Note the existing verdict from the file
3. Add any additional context you can provide based on your training knowledge
4. Flag any IOCs that warrant higher priority attention

Save the enriched analysis to outputs/[today's date]/ioc-analysis-scenario-01-02.md
```

**What to expect:** The agent reads the CSV, processes each row, and produces
an enriched analysis. It should flag `185.220.101.47` as particularly high
priority given its appearance in both scenarios and its Tor exit node status.

**What to check:** Does the agent stay honest about the limits of its knowledge?
It should note that its enrichment is based on training data and recommend
verifying against live threat intel sources for current verdicts.

## Exercise 5 — Edit a File

The agent can also edit existing files. This is useful for updating investigation
notes as an incident develops.

First, ask it to add a section to the file it created in Exercise 3:

```
Open outputs/[today's date]/investigation-notes-scenario-01.md

Add a new section called "Recommended Actions" at the bottom with three 
specific containment steps for this incident, based on what we know so far.
```

The agent will open the file, read its current contents, and append the new
section. Check the file again — the original content should be intact with the
new section added cleanly.

## Working With Different File Types

The lab contains several file formats you will encounter in real SOC work.
Here is how to reference each one effectively:

**JSON alert exports:**
```
Read lab/alerts/scenario-02-alert-phishing.json and summarize the attack chain.
```

**Raw log files (.log, .txt):**
```
Read lab/logs/linux-auth-srv-app-01.log and identify all failed SSH 
authentication attempts. List the source IPs and attempt counts.
```

**CSV IOC lists:**
```
Read lab/ioc/scenario-01-02-ioc-list.csv and tell me which IOCs 
are shared across both scenarios.
```

**CloudTrail JSON:**
```
Read lab/logs/cloudtrail-iam-scenario-02.json and identify the sequence 
of AWS API calls. Which ones failed and why?
```

## Output File Naming Convention

Consistent naming makes your outputs folder useful over time rather than
a pile of files you have to search through. Adopt this convention from the start:

```
outputs/YYYY-MM-DD/
├── triage-[scenario or alert ID].md         Alert triage summaries
├── investigation-notes-[scenario].md        Running investigation notes
├── ioc-analysis-[scenario].md               IOC enrichment results
├── sigma-[behavior-description].yml         Detection rules
├── ir-timeline-[incident-id].md             Incident response timelines
├── sir-[incident-id].md                     Security incident reports
├── notification-[incident-id].md            Stakeholder notifications
└── cti-[actor or campaign].md               Threat intelligence reports
```

When asking the agent to save files, always specify the full path:

```
Save this to outputs/2025-01-14/triage-scenario-01.md
```

Not just:
```
Save this somewhere
```

The more specific you are about output location, the easier it is to find
and reference your work later.

## Troubleshooting File Operations

**Agent says it cannot find a file:**
Check that you are running Claude Code from inside the `ai4soc/` directory,
not a parent folder. Run `pwd` in a terminal tab to confirm your working directory.

**Agent writes to wrong location:**
Be explicit in your instruction. Instead of "save this report", say
"save this report to outputs/2025-01-14/triage-scenario-01.md"

**Agent outputs to chat instead of saving to file:**
Add "save to file" explicitly. The agent defaults to displaying output
in the terminal unless you specify a file path.

**File created but content is wrong:**
Tell the agent directly: "The file you created at [path] has an issue — [describe
the issue]. Please fix it." The agent will re-read the file and correct it.
